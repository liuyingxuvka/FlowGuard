## Why

FlowGuard has accumulated strong specialist routes, field evidence, proof gates, and maintenance helpers, but AI-facing discovery still exposes too much as a flat surface. This makes ordinary maintenance feel heavy: existing features are present, yet some are not connected into a route chain that an AI can follow from intent to evidence to the next maintenance action.

This change makes FlowGuard lighter without weakening its safety contracts by turning self-maintenance into an explicit route graph, field layering, and model-backed maintenance loop.

## What Changes

- Add a FlowGuard self-maintenance mesh that coordinates route graph completeness, field lifecycle compression, public API slimming, structure split governance, validation mesh evidence, and final claim boundaries.
- Expand route-first discovery so every installed maintenance route can be found through a compact route group with trigger, minimal inputs, outputs, evidence owner, and next-action handoffs.
- Add lightweight AI entry profiles that expose minimal input shapes for common maintenance tasks while keeping full evidence fields available behind route-owned expansion.
- Fold repeated field/proof concepts into shared evidence references and field-layer profiles instead of forcing every route prompt to expose every field.
- Preserve public facades and existing behavior while allowing oversized review modules to move implementation details into clearer internal route-owned helpers.
- Connect SummaryReport, MaintenanceScan, ExistingModelPreflight, FieldLifecycleMesh, StructureMesh, ModelMesh, TestMesh, Model-Test Alignment, DevelopmentProcessFlow, Risk Evidence Ledger, and Closure Contract as one self-maintenance chain.
- Record validation, install sync, shadow workspace sync, and local git limitations as DevelopmentProcessFlow evidence rather than treating them as informal cleanup.

## Capabilities

### New Capabilities
- `flowguard-self-maintenance-mesh`: Coordinates FlowGuard's own route graph, field layering, structure governance, validation mesh, and closure evidence as a parent maintenance model.

### Modified Capabilities
- `flowguard-ai-entry-simplification`: Route-first AI entry must prefer compact route profiles over flat helper/API lists.
- `flowguard-global-routing`: Global routing must expose installed satellite routes as discoverable route groups with handoffs and evidence boundaries.
- `existing-model-preflight`: Existing model lookup must feed the self-maintenance mesh before new route boundaries are created.
- `development-process-flow`: Development evidence must track route graph, field, structure, validation, install, shadow sync, and peer-write invalidations.
- `model-test-alignment`: Alignment must consume field projections and self-maintenance obligations when claims bind model, code, and tests.
- `test-evidence-mesh`: Large or slow validation must be represented as parent/child evidence instead of background liveness.
- `structure-refactor-mesh`: Public entrypoint and facade parity must govern any internal module split.
- `hierarchical-model-mesh`: Parent confidence must depend on current child self-maintenance evidence when FlowGuard's own models are split.
- `risk-evidence-ledger`: Final confidence must surface self-maintenance gaps, stale evidence, and unsupported broad claims.
- `flowguard-closure-contract`: Guard-family child reports must close or scope the self-maintenance claim.
- `flowguard-codex-skill-satellites`: Installed skill route maps must stay aligned with the package route graph.

## Impact

- Affected code: public API grouping in `flowguard/__init__.py`, route/helper docs, template guidance, and tests that verify API surface and route discovery.
- Affected models: new self-maintenance FlowGuard artifact under `.flowguard/` plus route-specific checks.
- Affected docs/skills: route-first API docs, modeling protocol, installed skill hot paths, adoption logs, and OpenSpec artifacts.
- Affected validation: OpenSpec strict validation, focused unit tests, existing self-model checks, practical regression evidence, editable install verification, shadow workspace verification, and local git status checks where a git repository is available.
